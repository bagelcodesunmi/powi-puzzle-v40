# 포위 퍼즐 프로젝트 컨텍스트

## 프로젝트 개요
- **장르**: 턴제 전략 퍼즐 (포위 메커닉)
- **본질 기준**: 이 게임은 퍼즐 게임이다. 판 읽기, 수순 설계, 공간 제어, 포위 메커닉 이해를 우선한다.
- **플랫폼/표현 기준**: HTML 기반 2D 게임이다. 모든 기믹과 연출은 HTML 2D에서 명확하게 보이고 조작 가능해야 한다.
- **최적화 기준**: 새 기믹이나 표현을 검토할 때는 항상 성능/복잡도 대비 가치, 렌더링 비용, 계산량을 함께 본다.
- **현재 게임**: `powi-puzzle.html` (edge wall 기반, 레벨 소스는 `Level/levels.json`)
- 구 빌드(`포위퍼즐-v41.html` 메인 안정판, `powi-puzzle-v43.html` 뼈대)는 `archive/`로 이동 (2026-07-18)
- **작업 경로**: `~/Desktop/game-planning/포위퍼즐-v40/`

## 핵심 게임 규칙 (v40 기준)
- **용어(glossary)**: 화살 쏘는 주체·플레이어가 놓는 것 = **내블록**. "벽"은 경계벽(edge wall) 전용. 상세 `docs/glossary.md`.
- **배치**: 빈칸 전용, 내블록 HP 1 고정 (반격 1발 = 파괴 — HP 상승은 추후 아이템 시스템으로, 2026-07-19 Director. 용어는 내구도 대신 HP로 통일)
- **포획**: 영역 빈칸 0(완전 밀폐) → 내 화살 최대 매칭(내블록·적 좌상단 순) — 내블록당 1킬, 미조준 적 생존
- **반격**: 밀폐 영역의 **전투 속성** 적 전원(죽는 적 포함)이 최대 매칭으로 내블록에 1타씩
- **확산**: 예고(announced) 칸이 유효하면 그 칸, 무효면 최악 가정으로 비자살 프런티어 전체 분기
- **판정(v37 유예)**: 확산 레벨은 마지막 돌 후 적 턴 1회까지 실행 후 판정. 정지 레벨은 즉시 판정.
- **적 개체 속성 (2026-07-19, `docs/checklist-enemy-traits.md`)**: 적은 `{hp: 1|2, combat: bool}` 개체.
  레벨 `combat` 플래그는 기본값. 확산 자식은 hp1 + 혈통 전투 상속(혼합 인접은 전투 우선).
  hp2는 볼리 라운드당 1타 — 안 쏜 벽이 있으면 연타. **발사는 밀폐 "순간" 1회**: 볼리 후 생존한 밀폐 적은
  잠금 상태(재발사 없음), 영역이 다시 열리고 재밀폐되면 새 볼리 대상.
  데이터: `enemies: [{x,y,hp?,combat?}]` (게임 배열형은 `[x,y,hp?,combat?]`).
- **금 간 벽 (2026-07-20, `docs/checklist-cracked-walls.md`)**: 적만 부술 수 있는 1히트 경계벽
  (`crackedRightWalls`/`crackedDownWalls`). 인접 blob이 있으면 그 적 턴은 확산 대신 벽 파괴 (좌상단 우선 결정론,
  예고 표시, PRNG 소비 없음). 판당 ≤3. 일반 경계벽·내 돌은 계속 불가침 (비대칭). 데뷔 = 고수 구간(L71+).

## 변경 작업 원칙
- 기믹/규칙/초반 템포를 바꾸는 작업은 구현 전에 반드시 용어 정의와 기준선을 먼저 확정한다.
- 기존 체감에서 절대 안 깨야 하는 항목(예: `내 턴 끝나면 적이 바로 둔다`)은 먼저 문장으로 고정하고, 실험 중에도 유지 여부를 반복 확인한다.
- baseline 파일과 experiment 파일을 섞지 않는다. 실험은 반드시 복사본/별도 파일에서만 진행한다.
- 검증되지 않은 중간 산출물은 Director에게 완료물처럼 전달하지 않는다. 테스트 가능한지, 실제로 의도대로 동작하는지 확인된 것만 전달한다.
- 변경 전 확인은 `docs/change-checklist.md` 템플릿을 사용한다.

- **레벨 소스(단일)**: `Level/levels.json` — 편집 후 `node sync-levels.js`로 게임·툴에 반영 (`docs/tools.md` 참조)
- `level-lab.html` — 레벨 설계 & 검증 GUI 도구 (edge wall 배치·솔버·시드 뽑기·예산 지정·게임 파일 직접 저장 지원)

- **신규 레벨은 카탈로그에서 시작**: `catalog-viewer.html`(루트, 더블클릭 지원 — 데이터는 Catalog/) + `catalog-walls.js` —
  적1+벽≤2 전 배치의 min·예고봇 승률 전수 계산 DB. 워크플로·구조적 결론은 `docs/catalog-guide.md` 참조.
- 현재 게임에서 안 쓰는 것들은 전부 `archive/`로 이동: 구 도구·탐색 스크립트(budget-scan, find-*, solver-v40.js, seed-validator.js — edge wall 미지원), 구 프로토타입(v40~v42, v43 뼈대, v41 메인 안정판), `레벨-감정곡선.html`(구 15레벨 기준 시각화)

## 레벨 설계 불변 규칙
- **보드판은 절대 줄어들지 않는다** (2026-07-18 Director 지시): 레벨 진행에서 보드 크기는 단조 증가.
  한 번 5×5에 도달하면 이후 레벨은 전부 5×5 이상. 새 개념(전투 등) 도입 시에도 보드를 되돌리지 말고
  벽 프리셋으로 국소 난이도를 낮춰서 가르친다.
- **최대 보드는 6×6** (2026-07-19 Director 지시): 근거 — ① 적대 솔버 검증 가능선(min 확정은 8~9가 실질 상한,
  6×6은 벽 프리셋으로 min을 눌러야 검증 가능, 10×10은 검증 불가) ② 모바일 캔버스 520px 기준 칸 크기·오탭
  ③ "압축 퍼즐 + 정확한 예산" 정체성 (분산 지배 방지). 6×6은 피날레/보스 판으로 아껴 쓴다.
  스케일감이 더 필요하면 보드 확대가 아니라 적 수 증가(6×6 + 2~3기)로 해결한다.

## 레벨 데이터 형식
```js
{ board: N, enemies: [[x,y]], walls: [[x,y,hp]], schedule: [...], spread: bool, combat: bool, min: N }
```
- `board`: 보드 크기 (N×N)
- `enemies`: 적 위치
- `walls`: 벽 위치 + HP
- `schedule`: 확산 스케줄 (돌 수 배열)
- `spread`: 확산 레벨 여부
- `combat`: 반격 레벨 여부
- `min`: 클리어에 필요한 최소 돌 수 (예산)
