# 레벨 후보 카탈로그 가이드

> 대상: `catalog-walls.js`(생성·검색 CLI), `Catalog/catalog-viewer.html`(시각 탐색 GUI), `Catalog/catalog-b*.json`(데이터) — 툴·데이터는 `Catalog/` 폴더에 모여 있음
> 목적: 신규 레벨을 손으로 설계하는 대신, **미리 전수 계산된 경우의 수에서 조건 검색으로 시작**하기 위한 시스템
> 작성일: 2026-07-19 (Director 제안으로 도입 — decisions.md 2026-07-18 참조)

---

## 1. 무엇인가

"**적 1기 + edge wall 0~1개**"로 만들 수 있는 모든 배치를 보드별(3×3/4×4/5×5)로 전수 솔브해 둔 데이터베이스다.
대칭(회전·반전 8변형)으로 중복을 제거한 **대표형**만 담는다: 3×3 = 36항목, 4×4 = 102항목, 5×5 = 272항목.

각 항목이 담는 것:

| 필드 | 의미 |
|------|------|
| `board, combat, enemy, wall` | 배치 (wall: `{type:'r'\|'d', x, y}` 또는 null) |
| `min` | **적대 솔버 확정 최소 턴** — 최악 확산에서도 이 예산이면 클리어 가능. `null` = cap 초과/타임아웃 (5×5 정중앙 12개 = min 9 이상) |
| `bot` | **예고봇 승률(%)** — min 예산에서 800판 측정. 90%+ 튜토리얼급 / 40~80% 중간대 / 40% 미만 심화 (v43 가이드 §14-1) |
| `botMedian` | 봇이 이길 때 실제 사용한 돌 수 중앙값 (min보다 작으면 "요행 클리어" 존재 = 분산형) |
| `shieldedCorner` | 차폐 코너 여부 (§9-6 안티패턴 — 초반 레벨 금지, 검색 시 기본 제외) |
| `cap, timedOut` | 솔버 예산 상한·타임아웃 여부 (재생성 시 이어하기 판단에 사용) |

## 2. 도구 3종

### Catalog/catalog-viewer.html — 눈으로 탐색 (가장 먼저 열 것)
- **더블클릭(file://)으로 열림** — 같은 폴더의 `catalog-snapshot.js`(압축 스냅샷) 사용. 서버(localhost)로 열면 json 실시간 로드.
  복사·배포 시 두 파일을 함께 옮길 것. 스냅샷 크기는 뷰어 본체와 분리돼 있어 카탈로그가 커져도 HTML은 그대로.
- 필터: 보드 / 전투 / min / 예고봇 대역 / **벽 위치(적 인접·원거리·없음)** / 차폐 코너 포함. 정렬: min·예고봇.
- 카드 클릭 → **좌우/상하 미러 토글**(게임 미러링이 보여줄 4방향 미리보기) + **JSON 복사**.

### catalog-walls.js — 생성·검색 CLI
```bash
# 검색 (일상적 사용)
node catalog-walls.js --query 5 --combat true --bot-min 40 --bot-max 80   # 5×5 전투 중간대
node catalog-walls.js --query 4 --min 3                                   # 4×4 정확히 min 3

# 재생성 (규칙이 바뀌었을 때만 — 엔진은 level-lab 워커를 추출하므로 규칙과 자동 동기화)
node catalog-walls.js --build 5 --cap 20 --entry-timeout 300000
node catalog-walls.js --build 5 --cap 20 --shard 0/4    # 병렬 4프로세스 (0~3 각각 실행)
node catalog-walls.js --merge 5                          # 샤드 병합 + 뷰어 스냅샷 갱신
node catalog-walls.js --sync-viewer                      # 뷰어 스냅샷만 수동 갱신
```
- 이어하기 지원: 재실행 시 확정 항목은 스킵, `min: null` 항목은 cap이 커졌을 때만 재계산.

### Catalog/catalog-b{N}.json — 데이터 (커밋 대상)
재생성 비용이 크므로(5×5 심층 항목당 수십 초~수 분) 저장소에 커밋해 둔다.

## 3. 신규 레벨 제작 워크플로

1. **뷰어에서 조건 탐색** — 목표 min·예고봇 대역으로 필터 → 후보 카드 선택 → 미러 토글로 방향 결정 → JSON 복사
2. **정밀 재측정** — 카탈로그 봇 %는 800판(±3~4%p 노이즈). 채택 전 2,000판+ 재측정 필수:
   `node verify-level.js '<복사한 cfg>' --cap 6 --runs 2000 --exact` (또는 level-lab GUI)
3. **정확히 N턴 확인** — 학습 레벨이면 min−1 예산 우호 확산 5,000판 0승 확인 (§14-②)
4. **시드 추출·미러 검증** → `Level/levels.json` 반영 → `node sync-levels.js` → 실플레이 확인 (§14-③④)

## 4. 카탈로그가 알려준 구조적 사실 (2026-07-19 기준)

레벨 설계 시 헛수고를 줄여주는 전수 확인 결론들:

- **예고봇 90%+(튜토리얼급)는 대부분 min 1~3 밀착 벽 주머니에만 존재**한다. min이 깊으면서 승률이 높은 판은 벽 1개로는 안 나온다.
- **인접 벽 = 초반 문법, 원거리 벽 = 중후반 재료** (Director, 2026-07-19): 적의 면에 붙은 벽은 "이미 닫힌 면"으로
  즉시 읽히지만, 떨어진 벽은 의도가 나중에야 드러난다. 데이터로도 확인됨 — 90%+ 대역은 인접 벽 위주(18 vs 14,
  원거리 14개도 대부분 3×3 좁은 보드), 중간대는 원거리 위주(14 vs 111). 확정 레벨 L1~L10은 전부 인접 벽.
  → **학습 레벨을 고를 땐 뷰어의 "벽 위치: 적 인접" 필터를 켜고 시작**하고, 원거리 벽은 읽기 시험이 목적인 중후반에만 쓴다.
- **4×4에서 min 4 + 90%+는 존재하지 않음** (최고 81.8%, 요행 있음 / 정확히 4턴 중 최고 76%).
- **4×4에서 min 5 + 100%는 어떤 조합으로도 존재하지 않음** — 적1+벽2 전수(564대표형), 적2+벽≤1 전수(389대표형)까지 확인. 최고는 적 2기 (0,0)·(0,2)+오른벽(0,0)의 92%(요행 26%), 정확히 5턴 중엔 (0,0)·(0,1)+오른벽(0,0)의 86.5% (겹 벗기기 학습 구조).
- **전투(반격)는 min을 거의 안 바꾸고 예고봇 승률만 크게 떨어뜨린다** (예: 5×5 코너 51%→5%). 반격 = 턴 수가 아니라 수읽기 깊이 장치.
- **예산 슬랙(+1)은 전투 판 승률을 거의 못 올린다** (15→16% 수준). 난이도 조절은 벽 배치로 할 것. (예외: 비전투 L4처럼 min 수순이 심층 강제 수순인 경우엔 슬랙이 0%→100%로 뒤집기도 함)
- **5×5 정중앙 적은 min 9 이상** (12개 항목 타임아웃) — 심화 전용 재료.

## 5. 한계·확장

- 카탈로그 범위는 **적 1기 + 벽 0~1개**다. 벽 2개·적 2기는 필요할 때 스윕 스크립트로 확장한다
  (선례: 2026-07-19 L4 후보 탐색 — 세션 스크래치의 sweep2wall.js/sweep2enemy.js 패턴, 대칭 정규화 코드는 catalog-walls.js의 `canonical()` 재사용).
- 좌표는 대표형이므로 실제 배치 시 원하는 반전형으로 변환해 쓴다 (뷰어의 미러 토글이 계산해 줌).
- 게임 규칙이 바뀌면(포획·확산·반격) 카탈로그 전체가 무효 — level-lab 워커가 바뀐 시점에 `--build` 재실행.
