# 변경 전 확인 템플릿

> 목적: 기믹/규칙/레벨 흐름을 바꾸는 작업에서 의미 혼선과 기준선 파괴를 막기 위한 사전 확인 템플릿
> 작성일: 2026-07-17

---

## 프로젝트 고정 기준

작업 전 아래 3가지는 항상 이미 참이라고 가정한다.

- 이 게임은 **퍼즐 게임**이다
- 장르는 **턴제 전략 퍼즐**이다
- 표현 매체는 **HTML 기반 2D 게임**이다
- 새 기믹/연출/시스템은 항상 **최적화 관점**을 같이 검토한다

즉 변경 아이디어는 먼저 아래 질문을 통과해야 한다.

- 퍼즐로서 더 읽히는가?
- HTML 2D에서 명확하게 표현 가능한가?
- 성능/복잡도 대비 가치가 있는가?

---

## 짧은 실전 버전

작업 전에 아래 5줄만 먼저 채운다.

```text
1. 바꾸는 대상:
2. 핵심 용어 정의:
3. 절대 안 깨야 하는 기준선:
4. 이번에 바꾸는 축 1개:
5. 완료 기준:
```

---

## 상세 버전

### 1. 바꾸는 대상
- 무엇을 바꾸려는가?
- 예: 벽 시스템, 초반 레벨 구조, 턴 템포, UI 표현

### 2. 핵심 용어 정의
- 이 작업에서 중요한 용어는 정확히 무엇을 뜻하는가?
- 예:
  - `벽 = 칸 점유형` / `벽 = 칸 사이 경계형(edge wall)`
  - `공격 차단 포함` / `확산만 차단`

### 3. 절대 안 깨야 하는 기준선
- 이번 변경에서도 유지해야 하는 기존 동작은 무엇인가?
- 예:
  - `내 턴이 끝나면 적이 바로 둔다`
  - `v41 기존 레벨 체감 유지`
  - `기존 파일은 안 건드리고 복사본에서만 실험`

### 4. 이번에 바꾸는 축 1개
- 이번 작업에서 딱 무엇만 바꾸는가?
- 예:
  - 렌더링만
  - 확산 판정만
  - 레벨 1 데이터만

### 5. 실험 범위
- 어디까지가 실험이고, 어디는 손대지 않는가?
- 예:
  - `powi-puzzle-v43-edge-test.html`에서만 실험
  - L1만 수정
  - L2/L3는 그대로 유지

### 6. 완료 기준
- 무엇이 되면 이번 작업은 끝난 것으로 보는가?
- 예:
  - 벽이 블록 사이 선으로 보인다
  - 적이 wall 너머로 확산하지 않는다
  - 기존 초반 턴 템포가 유지된다

### 7. 초심자 첫 시선 / 첫 터치
- 처음 보는 사람이 어디를 먼저 볼 것 같은가?
- 모바일에서 어떤 칸이 가장 먼저 눌릴 것 같은가?
- 같은 퍼즐 논리라면 더 직관적인 입력 방향(예: 적의 위보다 아래)이 있는가?

### 8. 레벨 수정 범위 잠금
- 이번 수정의 기준 레벨은 무엇인가?
- 이번에 실제로 수정할 레벨은 무엇인가?
- 요청받지 않은 주변 레벨은 수정 금지로 잠갔는가?
- Director에게 전달하기 전, 이 결과가 실제로 테스트 가능하고 의도대로 동작하는지 확인했는가?
- 검증되지 않은 중간 상태를 완료물처럼 전달하지 않도록 잠갔는가?

형이 아니라 칸 사이 경계형(edge wall)
3. 절대 안 깨야 하는 기준선: 초반 레벨의 기존 턴 템포(내 턴 끝나면 적이 바로 둔다)
4. 이번에 바꾸는 축 1개: 벽 렌더링과 연결 판정만
5. 완료 기준: 벽이 선으로 보이고, wall 너머로 확산/공격이 안 된다
```

## 운영 원칙

- 애매하면 먼저 묻고, 명확해지면 작업한다
- baseline 파일과 experiment 파일을 섞지 않는다
- 한 번에 한 축만 바꾼다
- 체감 리듬 같은 기준선은 먼저 문장으로 고정한다
- 초반 튜토리얼은 퍼즐 논리뿐 아니라 첫 시선과 첫 터치 직관도 함께 검토한다
